## 概述

本文档基于现有系统分析和用户需求，制定了 IntelliSearch 的综合架构构建规划。规划重点关注安全机制、异常处理、事件交互等核心功能的实现，同时明确了各功能之间的依赖关系和实施优先级。

### 当前系统状态

#### 已实现功能 ✅

- **基础架构**：
  - MCP 工具调用框架（`tools/mcp_base.py`）
  - 智能体核心（`agents/mcp_agent.py`）
  - 日志系统（`core/logger.py`）
  - 配置管理（`config/config_loader.py`）

- **前端界面**：
  - CLI 界面（`cli.py` + `backend/cli_backend.py`）
  - Web 界面（`frontend/app.py`）
  - Rich UI 组件（`ui/` 目录）

- **异常处理体系**（2024-03-02 完成）：
  - 简化的异常类体系（6个严重级别）
  - ErrorCenter 错误中心和决策系统
  - ErrorCode 对象和 ErrorCodes 类
  - 完整的测试覆盖和文档

- **现有安全机制**：
  - 文件系统权限检查（`mcp_server/operate_file/security.py`）

#### 未实现功能 ❌

- **安全机制**：
  - Safety Agent（安全评估代理）
  - 安全规则引擎

- **交互机制**：
  - 事件管线（Event Pipeline）
  - UI 渲染引擎
  - 用户请求确认机制

## 详细功能规划

### 功能 1: 异常类体系

**状态**: ✅ 已完成（2024-03-02）

**优先级**: 🔴 高（整个系统的基础）

**实施阶段**: 阶段 1 ✅ 已完成

**功能描述**:
建立完整的异常类体系，提供统一的错误类型、错误码和上下文信息。

**异常类型**:

1. **基础异常** (`IntelliSearchError`)
   - 统一的异常基类
   - 包含错误码、消息、上下文
   - 支持异常包装

2. **安全异常** (`SecurityError` 及子类)
   - `PermissionDeniedError` - 权限拒绝
   - `InvalidPathError` - 无效路径
   - `DangerousOperationError` - 危险操作
   - `SensitiveDataError` - 敏感数据

3. **MCP 异常** (`MCPError` 及子类)
   - `MCPConnectionError` - 连接错误
   - `MCPToolNotFoundError` - 工具未找到
   - `MCPExecutionError` - 执行错误
   - `MCPTimeoutError` - 超时

4. **智能体异常** (`AgentError` 及子类)
   - `AgentInitializationError` - 初始化错误
   - `AgentExecutionError` - 执行错误

5. **配置和验证异常**
   - `ConfigurationError` - 配置错误
   - `ValidationError` - 验证错误

**集成点**:

- `MCPBase` - 在工具调用中抛出特定异常
- `MCPAgent` - 在智能体执行中捕获和处理异常
- `GlobalExceptionHandler` - 统一处理所有异常

**验收标准**:

- [ ] 覆盖所有主要错误场景
- [ ] 异常包含完整的上下文信息
- [ ] 支持异常链（异常包装）
- [ ] 提供友好的错误消息
- [ ] 支持错误码映射

---

### 功能 2: 全局异常处理器

**状态**: ✅ 已完成（2024-03-02）

**优先级**: 🔴 高

**实施阶段**: 阶段 1 ✅ 已完成

**功能描述**:
提供统一的异常捕获、分类、恢复和用户反馈机制。

**核心功能**:

1. **异常捕获和分类**
   - 捕获所有未处理的异常
   - 分类为不同类型（安全、连接、超时等）
   - 记录详细的异常日志

2. **恢复策略**
   - 自动重试（连接错误）
   - 降级处理（超时错误）
   - 用户确认（安全错误）
   - 终止执行（致命错误）

3. **用户友好消息**
   - 根据异常类型生成不同的消息
   - 使用表情符号和格式化文本
   - 提供恢复建议

4. **异常回调**
   - 支持注册特定异常的回调
   - 允许自定义处理逻辑

**集成点**:

- CLI 界面 - 在主循环中捕获异常
- Web 界面 - 在 API 路由中捕获异常
- MCPBase - 在工具调用中捕获异常

**验收标准**:

- [ ] 所有异常都被正确捕获
- [ ] 用户看到友好的错误提示
- [ ] 异常被正确记录到日志
- [ ] 支持不同的恢复策略
- [ ] 异常回调机制正常工作

### 功能 3: 安全协议层

**状态**: ❌ 未实现

**优先级**: 🔴 高

**实施阶段**: 阶段 2

**功能描述**:
在工具执行前进行安全检查，包括权限验证、参数净化、危险操作检测。

**核心组件**:

1. **安全验证器** (`SecurityValidator`)
   - 检查工具权限
   - 验证路径安全性
   - 检测危险命令
   - 净化敏感信息

2. **安全审计器** (`SecurityAuditor`)
   - 记录安全事件
   - 记录权限拒绝
   - 记录危险操作

3. **安全上下文** (`SecurityContext`)
   - 用户身份信息
   - 权限级别
   - 允许/禁止路径
   - 审计配置

**安全检查**:

- 路径遍历检测（`../../etc/passwd`）
- 系统目录访问（`/etc`, `/sys`, `/proc`）
- 危险命令检测（`rm -rf /`, `dd if=/dev/zero`）
- 敏感信息过滤（password, api_key, token）
- 权限级别验证

**集成点**:

- `MCPBase.execute_tool_calls()` - 在执行工具前验证
- 异常处理 - 抛出 `SecurityError` 异常

**验收标准**:

- [ ] 能够识别和阻止危险操作
- [ ] 权限检查正常工作
- [ ] 参数净化有效
- [ ] 安全事件被正确记录
- [ ] 不影响正常操作的性能

- 检测到风险操作时，抛出异常
- 通过全局异常处理器显示错误

**未来完整方案**:
在交互机制完成后，Safety Agent 可以：

- 检测到风险操作时，通过事件管线发送 `RequestUserConfirm`
- UI 渲染引擎渲染确认界面
- 用户做出决策后，事件管线返回结果
- Safety Agent 根据用户决策继续执行

### 2. 风险管理

**技术风险**:

- 事件管线架构复杂，可能需要多次迭代
- UI 渲染引擎需要同时支持 CLI 和 Web，兼容性挑战大

**缓解措施**:

- 采用渐进式开发，先实现核心功能
- 使用原型验证关键设计
- 保持代码模块化，便于调整

**进度风险**:

- 功能依赖多，某个功能延迟会影响后续功能
- 团队资源有限

**缓解措施**:

- 明确依赖关系，合理安排并行开发
- 优先实现高优先级功能
- 定期评估进度，及时调整计划
