# 错误中心处理器设计

## 一、核心定位

错误中心处理器是整个异常处理体系的中央枢纽，负责统一处理所有未在组件层捕获的异常。

**设计原则**：

- 唯一入口：所有未处理异常最终都到达这里
- 统一决策：基于统一的策略矩阵做决策
- 零业务逻辑：只做技术层面的处理，不做业务决策

## 二、核心职责

### 2.1 异常捕获与分类

**捕获范围**：

- 所有继承自 `IntelliSearchError` 的异常
- 通过 `OtherError` 包装的未知异常
- 未在组件层处理的任何异常

**分类依据**：

- 异常类型（error_code）
- 严重级别（severity）
- 上下文信息（context）

### 2.2 处理决策

**决策维度**：

| 维度                  | 说明         | 用途               |
| --------------------- | ------------ | ------------------ |
| error_code            | 错误码       | 确定错误类型和来源 |
| severity              | 严重级别     | 决定处理策略       |
| context.domain        | 领域标识     | 特定领域的特殊处理 |
| context.original_type | 原始异常类型 | 调试和日志         |
| 配置策略              | 全局配置     | 重试次数、超时等   |

**处理策略矩阵**：

| 级别     | 退出 | 阻断 | UI   | 日志     | 通知 MCP | 恢复     |
| -------- | ---- | ---- | ---- | -------- | -------- | -------- |
| FATAL    | ✓    | ✓    | 弹窗 | CRITICAL | ✗        | 无       |
| CRITICAL | ✗    | ✓    | 弹窗 | ERROR    | ✓        | 等待用户 |
| ERROR    | ✗    | ✓    | 警告 | ERROR    | ✓        | 上层决定 |
| WARNING  | ✗    | ✗    | 提示 | WARNING  | ✓        | 继续     |
| NOTICE   | ✗    | ✗    | 提示 | INFO     | ✗        | 等待响应 |
| INFO     | ✗    | ✗    | 无   | INFO     | ✗        | 静默     |

### 2.3 日志记录

**映射规则**：

- FATAL / CRITICAL → CRITICAL 级别日志
- ERROR → ERROR 级别日志
- WARNING → WARNING 级别日志
- NOTICE / INFO → INFO 级别日志

**日志内容**：

- 错误码和消息
- 严重级别
- 完整上下文（JSON 格式）
- 调用栈（开发模式）
- 原始异常信息（如果是 OtherError）

**日志目标**：

- FATAL / CRITICAL / ERROR → error.log
- WARNING → warning.log
- NOTICE / INFO → info.log

### 2.4 UI 触发

**触发原则**：

- 不直接与用户交互
- 只触发展示类事件
- 用户交互由事件系统负责

**事件类型**：

- FATAL / CRITICAL：`UIEventType.CRITICAL_DISPLAY`（弹窗展示）
- ERROR：`UIEventType.ERROR_DISPLAY`（顶部警告）
- WARNING：`UIEventType.WARNING_NOTIFICATION`（右上角通知）
- NOTICE：`UIEventType.NOTICE_DISPLAY`（内联提示）
- INFO：不触发 UI

**事件内容**：

- 错误消息
- 严重级别
- 建议操作（可选）
- 不包含任何交互元素

### 2.5 系统决策

**FATAL 级别**：

- 记录关键日志
- 触发 UI 弹窗
- 清理资源
- 优雅退出程序

**其他级别**：

- 记录日志
- 触发 UI（根据级别）
- 返回给调用方（MCP Agent 等）
- 不做退出决策

### 2.6 与 MCP 智能体集成

**通知规则**：

- CRITICAL / ERROR / WARNING：通知 MCP
- FATAL / NOTICE / INFO：不通知 MCP

**传递内容**：

- 异常基本信息（error_code, message）
- 严重级别
- 关键上下文（不包含敏感信息）

**MCP 的响应**：

- 根据错误类型决定重试、调整或终止
- 错误中心处理器不干预 MCP 的决策

## 三、处理流程

### 3.1 标准处理流程

```
1. 捕获异常
   ↓
2. 验证异常类型（必须是 IntelliSearchError 或其子类）
   ↓
3. 提取关键信息（error_code, severity, context）
   ↓
4. 根据严重级别确定处理策略
   ↓
5. 记录日志
   ↓
6. 触发 UI 事件（根据级别）
   ↓
7. 通知 MCP（如果需要）
   ↓
8. 返回给调用方或退出程序
```

### 3.2 特殊情况处理

**未知异常**：

- 应该已被组件层包装为 OtherError
- 如果仍是原生异常，包装为 OtherError
- 按 ERROR 级别处理

**多重异常**：

- 记录所有异常
- 以第一个异常为主要异常
- 其他异常作为附加上下文

**循环异常**：

- 检测异常重复出现
- 记录警告
- 按原策略处理，避免死循环

## 四、与其他系统的集成

### 4.1 日志系统

**集成方式**：

- 使用 Python 标准的 logging 模块
- 建立独立的 logger：`error_center`
- 配置不同的 handler 和 formatter

**日志格式**：

- 结构化日志（JSON）用于分析
- 可读日志（文本）用于调试
- 同时支持两种格式

### 4.2 事件系统

**集成方式**：

- 通过事件总线触发事件
- 只触发展示类事件，不触发交互类事件
- UI 层只负责展示，不负责交互

**事件隔离**：

- 错误中心处理器的事件类型与普通事件类型分开
- 使用独立的事件命名空间

### 4.3 MCP 智能体

**集成方式**：

- 通过返回值传递异常信息
- MCP 捕获返回的异常信息
- MCP 根据错误调整策略

**信息过滤**：

- 不传递敏感上下文信息
- 不传递完整的调用栈
- 只传递必要的错误信息

## 五、配置管理

### 5.1 可配置项

**日志配置**：

- 日志级别（INFO, WARNING, ERROR, CRITICAL）
- 日志路径
- 日志格式（JSON / 文本）
- 日志轮转策略

**UI 配置**：

- UI 触发阈值（最低显示的严重级别）
- 弹窗自动关闭时间
- 通知显示持续时间

**系统配置**：

- 异常重复检测阈值
- 最大异常上下文大小
- 是否启用调试模式

### 5.2 配置来源

**优先级**：

1. 环境变量
2. 配置文件（config.yaml）
3. 代码默认值

**热重载**：

- 支持配置热重载
- 不重启服务即可更新配置

## 六、性能考虑

### 6.1 性能目标

- 异常处理延迟 < 10ms
- 日志写入延迟 < 5ms
- UI 事件触发延迟 < 5ms

### 6.2 优化策略

**异步处理**：

- 日志写入使用异步方式
- UI 事件触发使用异步方式
- 不阻塞主流程

**缓存机制**：

- 缓存格式化后的日志
- 缓存常用的错误信息

**批量处理**：

- 批量写入日志
- 批量触发 UI 事件

## 七、错误恢复

### 7.1 恢复原则

- 不自动恢复，除非配置明确允许
- 恢复操作由上层（MCP、业务逻辑）决定
- 错误中心处理器只负责记录和展示

### 7.2 恢复触发

**配置驱动的自动恢复**：

- 仅在特定错误码允许
- 仅在特定严重级别允许
- 需要配置明确的恢复策略

**示例场景**：

- 网络超时：自动重试（配置：max_retries=3）
- 配置缺失：使用默认值（配置：fallback_to_default=true）

### 7.3 恢复监控

- 记录所有恢复尝试
- 统计恢复成功率
- 监控恢复对性能的影响

**告警方式**：
