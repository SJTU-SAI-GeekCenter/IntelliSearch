# 异常处理体系重构开发规划

## 一、项目背景

当前异常处理体系需要扩展以支持更细粒度的错误分级，并为未来的 ErrorAgent 功能做准备。核心目标是建立统一、可靠的错误处理机制，同时增强对 AI 幻觉的防护能力。

## 二、核心设计理念

### 2.1 审计者模式

- **核心思想**：引入独立的 ErrorAgent 作为审计者，切断主 AI 的上下文幻觉
- **工作原理**：
  - 主 AI 负责复杂推理（易产生幻觉）
  - ErrorAgent 负责简单的事实判断（几乎不会出错）
  - 一旦检测到危险模式，立即抛出异常强制中断

### 2.2 分层职责

| 层级       | 职责               | 处理策略           |
| ---------- | ------------------ | ------------------ |
| 组件层     | 只抛异常，不处理   | 零错误处理逻辑     |
| ErrorAgent | 监控审计，检测规则 | 抛出自定义错误     |
| 全全局层   | 统一处理所有异常   | 日志记录、UI 触发  |
| 事件系统   | 处理用户交互       | 完全独立于异常处理 |

### 2.3 可选性设计

- ErrorAgent 是可选的，不强制使用
- Agent 的安全功能也是可选的
- 适应不同风险等级的场景

## 三、核心改动

### 3.1 异常严重级别扩展

从原来的三级扩展为六级：

| 级别     | 说明                   | UI 展示           | 是否退出 | 是否阻断 |
| -------- | ---------------------- | ----------------- | -------- | -------- |
| FATAL    | 致命错误，立即终止程序 | 弹窗+详细错误     | ✓        | ✓        |
| CRITICAL | 严重错误，需用户干预   | 弹窗+建议操作     | ✗        | ✓        |
| ERROR    | 错误，记录返回         | 红色警告+详情     | ✗        | ✓        |
| WARNING  | 警告，不阻止流程       | 黄色提示          | ✗        | ✗        |
| NOTICE   | 通知，需用户注意       | 蓝色提示+操作按钮 | ✗        | ✗        |
| INFO     | 信息，仅记录日志       | 无                | ✗        | ✗        |

### 3.2 异常类更新

- 为所有异常类添加 `_get_severity()` 方法
- 将 `UnknownError` 重命名为 `OtherError`
- 支持自定义错误（继承 `OtherError`）

### 3.3 ErrorAgent 设计

**核心特性**：

- 在主 AI 执行实际操作前介入
- 简化的判断逻辑（无需理解意图）
- 强硬措施：命中规则立即抛出异常
- 内置最大循环次数限制

**自定义错误格式**：

```
{错误名称, 严重级别, 触发条件}
例如：{deleteAllMailError, CRITICAL, "即将删除所有邮件"}
```

## 四、处理策略矩阵

### 4.1 全局处理器决策维度

基于以下维度统一处理所有异常：

- 异常类型（error_code）
- 严重级别（severity）
- 上下文信息（context，包括 domain、original_type 等）
- 配置策略（重试次数、超时时间等）

### 4.2 ErrorAgent 双重处理机制

**路径 A：传给 Agent**

- 通过异常机制
- Agent 可选地硬编码处理
- 开发者实现补救措施

**路径 B：传给错误处理器**

- 通过全局捕获
- 触发 UI 展示（仅展示，不允许交互）
- 用户交互由 Agent 通过事件管线触发

**关键点**：

- UI 只负责展示，不负责交互
- 用户交互由 Agent 的补救措施触发
- 避免重复提示和冲突

## 五、实施计划

### 阶段一：核心实现（高优先级）

**目标**：建立基础的异常处理框架

**任务清单**：

1. **扩展异常严重级别**
   - 在 `ErrorSeverity` 枚举中添加 FATAL、CRITICAL、NOTICE
   - 为所有异常类实现 `_get_severity()` 方法
   - 按照讨论中定义的严重级别分配

2. **重命名和扩展异常类**
   - 将 `UnknownError` 重命名为 `OtherError`
   - 增强 `from_exception` 方法，支持 `domain` 参数
   - 定义自定义错误的错误码格式：`OTH_###`

3. **实现全局异常处理器**
   - 创建中央异常处理入口
   - 实现基于严重级别的处理逻辑
   - 集成日志系统（映射到 logging 模块）
   - 实现 `get_log_level()` 方法

4. **更新 UI 事件系统**
   - 定义只读展示事件类型（如 `INFO_DISPLAY`）
   - 确保错误处理器触发的 UI 不包含交互元素
   - 验证与现有事件系统的兼容性

**预期产出**：

- 完整的异常类型体系
- 可工作的全局异常处理器
- 日志集成
- 基础的 UI 展示

### 阶段二：ErrorAgent 实现（中优先级）

**目标**：实现审计者模式的 ErrorAgent

**任务清单**：

1. **ErrorAgent 核心逻辑**
   - 设计监控接口
   - 实现规则检测引擎
   - 实现循环次数限制机制（默认 3 次）

2. **规则定义系统**
   - 设计规则格式（JSON 或 YAML）
   - 实现规则加载和解析
   - 建立错误码注册表（避免冲突）

3. **集成到执行流程**
   - 在 MCP Agent 执行操作前插入监控点
   - 实现异常抛出机制
   - 测试与现有流程的兼容性

**预期产出**：

- 功能完整的 ErrorAgent
- 规则定义文档和示例
- 集成测试

### 阶段三：优化和增强（低优先级）

**目标**：提升性能和开发体验

**任务清单**：

1. **Token 优化**
   - 提供精简的规则模板
   - 实现规则缓存机制
   - 优化检测逻辑

2. **开发者工具**
   - 提供异常处理模板
   - 提供规则定义向导
   - 实现调试和监控工具

3. **文档和示例**
   - 编写完整的 API 文档
   - 提供最佳实践指南
   - 提供常见场景的示例代码

**预期产出**：

- 性能优化报告
- 开发者工具集
- 完整的文档体系

## 九、讨论要点

### 9.1 需要确认的问题

1. **错误码分配**：是否需要统一的错误码注册表？
2. **规则定义格式**：JSON、YAML 还是其他格式？
3. **循环限制**：默认 3 次是否合适？是否需要可配置？
4. **Token 预算**：ErrorAgent 是否需要单独的 token 预算？

### 9.2 技术决策

1. **ErrorAgent 实现**：基于规则还是基于模型？（当前设计基于规则）
2. **监控粒度**：监控整个对话还是仅监控操作？（当前设计监控操作）
3. **恢复机制**：是否需要自动恢复机制？（当前设计不包含）

### 9.3 优先级讨论

1. 阶段二（ErrorAgent）是否可以延后？
2. Token 优化是否应该提前？
3. 是否需要先做概念验证（PoC）？
